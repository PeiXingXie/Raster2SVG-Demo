# 任务书：基于 Planner + ReAct 范式的 Raster 图像转 SVG 系统

## 1. 任务名称

**Raster 图像到 SVG 的自动化转换 Agent Demo**

---

## 2. 任务背景

现有 raster 图像，例如 PNG、JPG、截图、信息图、流程图、图表、页面设计稿等，通常难以直接编辑。目标是构建一个 Agent 系统，将输入的 raster 图像自动转换为结构化、可编辑、可检查、可修复的 SVG 文件。

该系统不追求像素级完全复刻，而是优先保证：

1. **整体布局一致**
2. **主要内容完整**
3. **图形元素可编辑**
4. **文本尽量保留为文本**
5. **流程图、图表等结构性内容可语义化表达**
6. **复杂图像允许使用合理占位或色块近似**

系统应采用 **Planner + ReAct** 的任务范式：

- **Planner** 负责全局任务规划、要求拆解、区域划分、任务调度和最终合并。
- **ReAct Worker** 负责单个区域内的观察、识别、生成、检查和修复。

---

## 3. 总体目标

构建一个能够将 raster 图像转换为 SVG 的 Agent Demo 系统。用户输入一张 raster 图像后，系统应自动完成以下工作：

1. 理解用户对 SVG 转换的要求。
2. 将抽象要求转化为可判别的 checklist。
3. 识别完整图像的 layout，并划分为多个 region。
4. 为每个 region 生成对应的 SVG 骨架。
5. 按 region 分配子任务。
6. 对每个 region 执行对象识别、SVG 生成、检查和修复。
7. 将各 region 的 SVG 结果合并为最终 SVG。
8. 输出最终 SVG、区域信息、检查结果和已知限制。

---

## 4. 目标产物

系统最终应输出以下内容：

```text
1. 最终 SVG 文件
2. 图像区域划分结果
3. 区域级 SVG 生成结果
4. checklist 及其检查结果
5. 对象识别摘要
6. 修复记录
7. 已知限制说明
```

其中最终 SVG 应满足：

```text
可渲染
可编辑
结构清晰
包含 region 分组
包含必要注释
尽量语义化表达主要内容
```

---

# 5. 功能范围

## 5.1 输入功能

系统应支持用户输入：

```text
1. 一张 raster 图像
2. 可选的用户要求
```

用户要求可以包括：

```text
转换为 SVG
尽量保留原始布局
保留可识别文本
图表需要生成坐标轴和数据近似
流程图需要保留框、线、箭头和标签
复杂图像可用色块替代
不嵌入原始位图
输出可编辑 SVG
```

---

## 5.2 Requirement Planning 功能

系统应将用户的抽象要求转化为可判断、可追踪、可验收的 checklist。

Checklist 应覆盖：

```text
1. 全局布局要求
2. 区域完整性要求
3. SVG 结构要求
4. 文本保留要求
5. 图标表达要求
6. 流程图表达要求
7. 连接线与箭头要求
8. 图表表达要求
9. 复杂图像替代要求
10. SVG 合法性要求
11. 最终结果可渲染要求
```

Checklist 中每一项应至少包含：

```text
编号
检查层级
检查标准
是否必须
检查结果
失败原因，可选
修复建议，可选
```

示例：

```text
C1：最终 SVG 的画布尺寸和原图比例一致。
C2：所有主要 layout region 都应在 SVG 中有对应分组。
C3：可识别文本应尽量以 <text> 表示。
C4：流程图中的框、线、箭头应使用 SVG 基础图形表示。
C5：图表应包含坐标轴、标签和近似数据图形。
C6：复杂自然图像或二维码可使用色块占位。
C7：最终 SVG 应为合法 XML 且可正常渲染。
```

---

## 5.3 Layout 识别与 Region 划分功能

系统应对完整图像进行 layout 分析，识别主要区域，并输出 region 划分结果。

每个 region 应包含：

```text
region_id
空间坐标
宽度
高度
简要描述
预期内容类型
处理优先级
处理状态
```

Region 类型可以包括：

```text
标题区域
正文区域
图标区域
流程图区域
连接线区域
图表区域
复杂图像区域
页脚区域
背景区域
混合内容区域
```

输出示例：

```json
{
  "region_id": "r2_main_flow",
  "bbox": {
    "x": 40,
    "y": 140,
    "width": 620,
    "height": 480
  },
  "description": "Main flowchart region with boxes and arrows.",
  "expected_objects": ["flow", "lines", "text"],
  "priority": 2
}
```

---

## 5.4 SVG Template 生成功能

系统应基于完整图像尺寸和 region 划分结果，生成全局 SVG template。

SVG template 应包含：

```text
SVG 根节点
画布尺寸
viewBox
全局样式区域
公共定义区域
每个 region 对应的 <g> 分组
每个 region 的注释
每个 region 的坐标信息
```

Template 不要求包含最终对象细节，但应为后续 region 填充提供结构。

示例：

```xml
<svg width="1200" height="800" viewBox="0 0 1200 800">
  <!-- Region r1_header: title area, bbox=(0,0,1200,120) -->
  <g id="r1_header" data-bbox="0,0,1200,120"></g>

  <!-- Region r2_main_flow: flowchart area, bbox=(40,140,620,480) -->
  <g id="r2_main_flow" data-bbox="40,140,620,480"></g>
</svg>
```

# 6. 子任务分配要求

系统应按 region 划分生成子任务。每个 region 子任务应包含：

```text
region_id
region 坐标
region 描述
region 对应的图像内容
region 对应的 SVG 骨架
预期对象类型
region 级 checklist
处理预算
输出要求
```

子任务应支持独立执行，并最终可被合并到全局 SVG。

---

# 7. Region 处理功能

每个 region 应通过 ReAct loop 进行处理。每个 region 的处理过程应包含以下阶段：

```text
1. 获取 region 对应的子图像
2. 识别 region 内的对象
3. 识别对象类型和空间位置
4. 生成对象级 SVG
5. 组合为 region SVG
6. 检查 region SVG 是否满足要求
7. 对不满足要求的对象或区域进行修复
8. 达到要求或预算上限后结束该 region
```

---

## 7.1 子图像获取

系统应根据 region 坐标获取对应的子图像，用于后续局部识别和生成。

子图像应对应原图中的特定区域，并保留其在全局图像中的坐标关系。

---

## 7.2 Object 识别功能

系统应识别每个 region 内部的主要 object，并输出对象清单。

每个 object 应包含：

```text
object_id
object 类型
空间坐标
尺寸
简要描述
所属 region
可选文本内容
可选样式提示
可选连接关系
```

Object 类型包括：

```text
icon
text
flow
lines
graph
fig
```

含义如下：

| 类型 | 说明 |
|---|---|
| `icon` | 图标、符号、装饰性小图形 |
| `text` | 可识别文本 |
| `flow` | 流程框、节点、步骤框、决策框 |
| `lines` | 连接线、箭头、分隔线、边框线 |
| `graph` | 柱状图、折线图、饼图等数据图表 |
| `fig` | 复杂图像、自然图像、二维码、难以矢量化的复杂内容 |

---

## 7.3 Object SVG 生成要求

系统应根据 object 类型生成对应 SVG 表达。

### 7.3.1 icon

图标应尽量使用可编辑 SVG 基础图形表达，例如：

```text
path
circle
rect
polygon
line
```

不要求像素级还原，但应保持语义和视觉近似。

---

### 7.3.2 text

可识别文本应尽量使用 SVG `<text>` 元素表达。

应尽量保留：

```text
文字内容
文字位置
字号近似
颜色近似
粗细近似
对齐方式近似
```

---

### 7.3.3 flow

流程图对象应使用可编辑图形表达，包括：

```text
矩形
圆角矩形
菱形
圆形
标签文本
连接线
箭头
```

流程图应尽量保持：

```text
节点位置
节点顺序
连接关系
箭头方向
标签内容
```

---

### 7.3.4 lines

连接线、箭头、分隔线应使用：

```text
line
polyline
path
marker
```

应尽量保持：

```text
起点
终点
方向
折线路径
箭头样式
线条颜色
线条粗细
```

---

### 7.3.5 graph

图表对象应被语义化转换，而不是只作为普通图片处理。

图表 SVG 应尽量包含：

```text
坐标系
横轴
纵轴
刻度
标签
图例
数据点
柱体
折线
区域
标题
```

图表中的数据可以是近似值，但应保持视觉趋势和相对关系。

---

### 7.3.6 fig

复杂图像、自然图像、二维码、照片等难以合理矢量化的内容可以使用色块或占位图形替代。

替代结果应包含：

```text
占位边界
主色块
简要注释
必要说明
```

例如：

```text
该区域为复杂自然图像，已用主色块占位。
```

---

# 8. Region 检查与修复功能

## 8.1 Region 检查

每个 region 生成后，系统应检查其是否满足对应 requirement 和 checklist。

检查内容包括：

```text
1. region 是否存在对应 SVG 分组
2. region 内容是否在指定坐标范围内
3. 主要 object 是否已生成
4. object 类型是否正确
5. 文本是否尽量保留
6. 流程图结构是否完整
7. 连接线和箭头是否完整
8. 图表是否包含坐标系和数据内容
9. 复杂图像是否使用合理占位
10. region SVG 是否合法
11. region 视觉效果是否基本符合原图
```

---

## 8.2 检查结果

检查结果应包含：

```text
状态
得分
通过项
失败项
问题严重程度
问题描述
问题位置
修复建议
```

状态可以包括：

```text
passed
failed
partially_passed
budget_exhausted
```

---

## 8.3 Region 修复

对于检查不通过的 region 或 object，系统应进行修复。

修复对象可以是：

```text
缺失对象
错误对象类型
错误文本
错误坐标
错误颜色
缺失箭头
缺失图表轴线
元素溢出 region
SVG 语法错误
```

修复应遵循：

```text
1. 优先修复高严重度问题
2. 优先修复结构性问题
3. 优先修复影响语义的问题
4. 在预算限制内进行
5. 修复后重新检查
```

---

## 8.4 预算限制

每个 region 的处理和修复应受预算限制。

预算可以包括：

```text
最大迭代次数
最大检查次数
最大修复次数
最大工具调用次数
最大处理时间
```

当 region 达到预算上限时，应停止当前 region 的处理，并记录状态和已知问题，然后继续处理下一个 region。

---

# 9. 全局合并功能

当所有 region 处理完成或达到预算限制后，系统应将各 region 结果合并为最终 SVG。

合并时应保证：

```text
1. 每个 region SVG 被放入对应的全局 <g> 分组
2. region 顺序与视觉层级合理
3. 坐标系统一
4. 样式定义统一
5. 公共 marker、clipPath、style 去重
6. object id 不冲突
7. SVG 结构合法
8. 最终结果可渲染
```

---

# 10. 最终检查功能

最终 SVG 生成后，系统应执行全局检查。

全局检查应覆盖：

```text
1. SVG 是否是合法 XML
2. SVG 是否可渲染
3. 画布尺寸是否正确
4. viewBox 是否正确
5. 所有主要 region 是否存在
6. 所有高优先级 region 是否已处理
7. 主要文本是否已保留
8. 主要图形结构是否已保留
9. 图表是否具备基本语义结构
10. 复杂图像是否有占位说明
11. 是否存在明显元素溢出
12. 是否存在 id 冲突
13. 是否存在不允许的 raster 嵌入
```

---

# 11. 操作流程

用户侧操作流程如下：

```text
1. 上传 raster 图像。
2. 可选输入转换要求。
3. 点击开始转换。
4. 系统显示任务解析结果和 checklist。
5. 系统显示 layout region 划分结果。
6. 系统逐 region 执行转换。
7. 系统显示每个 region 的处理状态。
8. 系统显示检查和修复记录。
9. 系统合并并生成最终 SVG。
10. 用户预览最终 SVG。
11. 用户下载最终 SVG。
12. 用户查看转换报告。
```

# 14. 非目标范围

本任务书不要求实现以下能力：

```text
1. 像素级完全一致的 SVG 复刻
2. 对复杂自然图像进行真实高精度矢量化
3. 对二维码进行可扫描重建
4. 对所有字体进行完全匹配
5. 对所有颜色、阴影、渐变进行精确还原
6. 对任意复杂图表进行精确数据还原
7. 自动生成生产级设计文件
8. 完全无人工检查的高保真交付
```

---

# 15. 成功标准

系统 demo 被认为成功，应满足以下标准：

```text
1. 能接收 raster 图像输入。
2. 能生成明确的转换 checklist。
3. 能识别并输出主要 layout region。
4. 能生成包含 region 分组和注释的 SVG template。
5. 能逐 region 识别 object。
6. 能按 icon/text/flow/lines/graph/fig 分类 object。
7. 能为主要 object 生成 SVG 表达。
8. 能检查 region 是否满足要求。
9. 能对不合格 region 或 object 进行至少一次修复。
10. 能在预算限制下继续处理后续 region。
11. 能合并各 region 为最终 SVG。
12. 最终 SVG 可渲染、结构清楚、主要内容可编辑。
13. 能输出转换报告和已知限制。
```

```text
5. 文本准确性

```

# 18. 一句话概括

本任务的目标是构建一个基于 **Planner + ReAct** 的 Raster-to-SVG Agent Demo：系统先将转换要求拆解为可验收 checklist，再进行图像 layout region 划分，随后逐 region 识别对象、生成 SVG、检查和修复，最后合并为结构清晰、可编辑、可渲染的最终 SVG。
