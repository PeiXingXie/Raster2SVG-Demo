## What This README Covers

This document focuses on:

- what the desktop shell is responsible for

- how to debug desktop-specific failures

## Role of the Desktop Client

The project keeps the FastAPI-hosted web frontend for:

- development
- debugging
- monitoring

The desktop client now uses a desktop-specific frontend shell.

Instead:

- FastAPI serves both the web development frontend and the desktop frontend
- the browser opens the web/development frontend at `/`
- Electron opens the desktop product shell at `/static/desktop.html`
- both shells reuse the same backend APIs and shared frontend modules

## What the Desktop Shell Needs

Required:

- a healthy backend service
- Node.js 20+ unless dependencies are already bundled

Usually the desktop shell points the backend health check to:

- `http://127.0.0.1:8120/`

Electron then opens:

- `http://127.0.0.1:8120/static/desktop.html`

Or a custom value from:

- `RASTER_SVG_FRONTEND_URL`
- `.runtime_startup.env` via `APP_HOST` and `APP_PORT`
- `.env` via `APP_HOST` and `APP_PORT`

## Recommended Use During Development

For full-stack local development, prefer the root startup entrypoint.

### Windows

```powershell
powershell -ExecutionPolicy Bypass -File .\start-dev.ps1 -Desktop
```

### macOS

```bash
chmod +x start-dev.sh
./start-dev.sh --desktop
```

### Linux

```bash
chmod +x start-dev.sh
./start-dev.sh --desktop
```

This path:

- starts or bootstraps the backend
- uses backend bootstrap mode `if-needed` by default through the root startup scripts
- starts or bootstraps desktop dependencies
- launches Electron against the desktop frontend shell

For the full developer workflow around Conda, `.env`, and backend startup, return to [README.developer.md](../README.developer.md).

## Manual Desktop Startup

Use this path when:

- the backend is already running
- you only want to debug desktop startup
- you want to override the service URL manually

### Step 1: Bootstrap desktop dependencies

#### Windows

```powershell
cd desktop
powershell -ExecutionPolicy Bypass -File .\bootstrap.ps1
```

#### macOS

```bash
cd desktop
chmod +x bootstrap.sh start-desktop.sh
./bootstrap.sh
```

#### Linux

```bash
cd desktop
chmod +x bootstrap.sh start-desktop.sh
./bootstrap.sh
```

What this does:

- checks Node.js version
- prefers packaged runtime files when provided
- otherwise uses system `node` and `npm`
- installs Electron dependencies when missing

### Step 2: Start the desktop shell

#### Windows

```powershell
cd desktop
powershell -ExecutionPolicy Bypass -File .\start-desktop.ps1
```

Override the service URL:

```powershell
$env:RASTER_SVG_FRONTEND_URL="http://127.0.0.1:8120/"
powershell -ExecutionPolicy Bypass -File .\start-desktop.ps1
```

#### macOS

```bash
cd desktop
./start-desktop.sh
```

Override the service URL:

```bash
RASTER_SVG_FRONTEND_URL="http://127.0.0.1:8120/" ./start-desktop.sh
```

#### Linux

```bash
cd desktop
./start-desktop.sh
```

Override the service URL:

```bash
RASTER_SVG_FRONTEND_URL="http://127.0.0.1:8120/" ./start-desktop.sh
```

What this does:

- resolves the service URL
- waits briefly for backend health through `/health`
- warns if the backend is still unreachable after the wait window
- launches Electron against the desktop frontend shell

## Service URL Resolution

The desktop startup logic resolves the service URL in this order:

1. explicit `RASTER_SVG_FRONTEND_URL`
2. `.runtime_startup.env` values such as `APP_HOST` and `APP_PORT`
3. `.env` values such as `APP_HOST` and `APP_PORT`
4. default `http://127.0.0.1:8120/`

If the resolved URL points to the service root, Electron automatically loads `/static/desktop.html`.

`.runtime_startup.env` is generated by the backend or developer startup scripts after port conflict handling completes.
This keeps the desktop shell aligned with the actual runtime port even when startup auto-switches away from the default port.
It does not overwrite your long-term `.env` defaults.

## Runtime Resolution

Desktop bootstrap prefers, in order:

1. explicit arguments such as `-NodeExe` or `NODE_EXE`
2. environment overrides such as `DESKTOP_NODE_EXE`
3. packaged runtime files under `desktop/runtime/node`
4. system `node` / `npm`

## Desktop-specific Troubleshooting

### Node.js or npm is missing

Check:

- Node.js 20+
- packaged runtime under `desktop/runtime/node`
- packaged `desktop/node_modules`

### Electron installs but remains incomplete

Symptoms:

- `npm install` succeeds
- `desktop/node_modules/electron` exists
- desktop bootstrap still reports that Electron is incomplete

Current Windows bootstrap now tries three recovery steps automatically:

1. `npm rebuild electron`
2. clean Electron package artifacts and reinstall
3. repair Electron from the local Electron cache zip when available

If it still fails, check:

- `C:\Users\<you>\AppData\Local\electron\Cache`
- antivirus or file-lock interference during unzip
- whether switching to Node.js 20 LTS or 22 LTS improves Electron installation stability

### Desktop starts but backend is unreachable

Check:

- backend is running
- `http://127.0.0.1:8120/health`
- `RASTER_SVG_FRONTEND_URL`
- `.env` values for `APP_HOST` and `APP_PORT`

The desktop startup scripts now wait for backend health before launching Electron, so a short delay before the window loads is expected when the API is still starting.

### Backend works in the browser but not in Electron

Check:

- service URL override points to the same backend URL
- no firewall or local binding issue prevents Electron from reaching the backend
- backend is listening on the address you expect

## Cross-platform Scope

This shell is intended to work on:

- Windows
- macOS
- Linux

without maintaining separate frontend implementations.
